The last reorganization has failed and you are trying to execute a different reorganization

Executing a reorganization the following error is displayed:

The last reorganization has failed and you are trying to execute a different reorganization.
Unexpected errors may occur if you don't try to finalize previous reorganization before running this one.
If you want to run this reorganization anyway, use 'ignoreresume' parameter.

Cause: A reorganization (we will name it reorgStep1) was executed and failed; then some change was done to the GeneXus Database model (surely trying to' fix' the reorgSetup1 failure). A new reorganization (we will name it reorgStep2) is executed and the detailed error occurs.

The GeneXus reorganization executing the reorgStep2 could detect an inconsistency (due to the reorgStep1 previous reorganization error) during the schema verification (done to avoid failures) and the error message is displayed.

Solution: There is no recipe for this error because it depends on several factors; check the following guidelines:

When a reorganization fails (in this case both reorgStep1 and reorgStep2 failed) the following options should checked:

  • Try to understand why the reorganization failed, solve the problem trying not to modify the GeneXus database model and run again the reorganization.
    In this case several reorganization generations will execute the same GeneXus Code; a sample of this error is login or connection problem with the DBMS, generally you modify the Database preferences and the reorganization is regenerated and executed until succeeded.
  • Figure out why the reorganization failed, modify the GeneXus database model to solve the problem and execute the reorganization again.
    In this case the reorganization may detail the above error because there is a mismatch between de GeneXus Schema model and the database one; to solve the problem you could:
    • Use the -ignoreresume parameter in the Reorganization Options property to bypass the default validation done by GeneXus and run the reorganization process from the very beginning.
      If the reorgStep1 reorganization did a change in the database the -ignoreresume parameter, your reorganization may still not work because the target database has some change that GeneXus is not aware of; you will have to restore a Database Backup and rerun the reorganization.
